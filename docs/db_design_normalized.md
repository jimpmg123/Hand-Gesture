# Normalized Database Design (3NF) - Travel From Photo

This document describes the recommended normalized PostgreSQL schema for the current Travel From Photo project. It is based on the existing codebase, the current frontend/backend structure, and the planned Travelize and AI Trip Journal features.

## users
```sql
id BIGINT PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
first_name VARCHAR(100) NOT NULL,
last_name VARCHAR(100) NOT NULL,
user_id VARCHAR(50) UNIQUE NOT NULL,
email VARCHAR(255) UNIQUE NOT NULL,
password_hash VARCHAR(255),
auth_provider VARCHAR(20) NOT NULL DEFAULT 'local',
google_sub VARCHAR(255) UNIQUE,
role VARCHAR(20) NOT NULL DEFAULT 'traveler',
is_active BOOLEAN NOT NULL DEFAULT TRUE,
created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
```

## user_privacy_settings
```sql
id BIGINT PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
user_id BIGINT UNIQUE NOT NULL,
allow_admin_image_access BOOLEAN NOT NULL DEFAULT FALSE,
gallery_visibility VARCHAR(20) NOT NULL DEFAULT 'private',
allow_profile_discovery BOOLEAN NOT NULL DEFAULT FALSE,
created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
```

## uploaded_images
```sql

owner_user_id BIGINT NOT NULL,

updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
```

## Notes
- **Authentication**: `users` supports both local auth and Google OAuth through `password_hash`, `auth_provider`, and `google_sub`.
- **Privacy Control**: `user_privacy_settings` keeps image visibility and admin-access rules outside the main user table.
- **Image Normalization**: raw file data lives in `uploaded_images`, exact EXIF data lives in `image_exif_metadata`, and inferred AI output lives in separate analysis tables.
- **Candidate Search**: one image can have many place candidates, so `image_place_candidates` is separated from `places`.
- **Gallery Structure**: gallery groups and their images are separated by `gallery_group_images` because one group contains many images and ordering may matter.
- **Travelize Design**: generated itineraries are stored in `travel_plans`, split into `travel_plan_days`, then ordered `travel_plan_items`.
- **AI Trip Journal Design**: journal images, inferred visit segments, generated entries, and cached weather are stored separately so exact and estimated information do not get mixed together.
- **3NF Goal**: each table stores one main kind of fact, with relationships represented by foreign keys instead of duplicated columns across many tables.
